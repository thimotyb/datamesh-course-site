1 Platform Team Access

This module presents a practical Google Cloud architecture for a self-serve data mesh platform. The design targets a middle ground between quick onboarding and meaningful domain autonomy. The operating model includes data-producing teams, a platform team, and data-consuming roles. White team produces and consumes analytical data products. Gray team owns the platform template and enablement workflow. BigQuery provides scalable analytical storage and SQL access. Cloud Storage enables object-based upload and file input patterns. Service accounts enforce access boundaries at product level. The platform interface is a versioned repository containing documentation and reusable templates. The platform kernel is a configurable Terraform template used to provision product stacks.

1.1 Pipeline Producers Products

This baseline keeps maintenance overhead low while still giving producers repeatable setup paths. Each data product is implemented as a pipeline and an owned BigQuery dataset with controlled access. A Dataflow pipeline can ingest from Pub/Sub, BigQuery, Cloud Storage, or upstream products. Output permissions are scoped so each consumer receives only the required data surfaces. The architecture supports push ingestion through file uploads and pull ingestion through pipelines. Push input can start from a Cloud Storage bucket with validated schemas and transformation rules. Pull input can combine upstream datasets and events to produce derived analytical products.

Producers retrieve the latest template, provision a stack, and configure pipeline sources. Producers define service-account policies for approved consumers and systems. Consumers discover products via SQL browsing and request additional access from owners.

2 Data Governance Metadata

This flow enables fast delivery while keeping ownership and authorization explicit. The baseline platform can be expanded incrementally without replacing the core provisioning model. Catalog integration can improve product discoverability and metadata quality. Log-derived usage metrics can surface popular datasets and frequent access patterns. Central quality checks can publish results to improve trust in product consumption. A lightweight GUI can simplify onboarding metadata and reduce manual errors. Dataplex is a unified data management and governance service for distributed data estates on Google Cloud. In a data mesh context, Dataplex helps domain teams keep decentralized ownership while applying consistent governance policies and metadata controls.

Cataloging and discovery: data assets are indexed and searchable across storage and analytics systems. Governance at scale: policy management and quality controls can be enforced consistently across domains.

2.1 Data Products Governance

Operational visibility: metadata, lineage, and quality signals improve trust and reuse of domain data products. Domain autonomy with standards: each team can manage products independently while complying with shared controls. A practical implementation path is available in the Google reference repository GoogleCloudPlatform/data-mesh-demo, which demonstrates domain-oriented data products, shared platform services, and governance automation patterns. The Google Cloud data mesh reference architecture emphasizes decentralized ownership with standardized interoperability. Domain-oriented ownership: each domain team owns its data products end to end, including quality and lifecycle. Data as a product: products must be discoverable, understandable, accessible, and governed for reuse.

Self-service platform: shared capabilities reduce repeated engineering effort across domains. Federated governance: common policies are centrally defined and computationally enforced. In practice, Dataplex supports this model by combining cataloging, governance, metadata management, and policy-driven controls. Automated checks and catalog workflows provide a practical entry point for federated governance.

3 Access Views Authorized

Ownership remains effective only if access control and policy boundaries are enforceable. Governance automation should avoid central bottlenecks and preserve team delivery flow. Strengths: low maintenance, fast integration of native services, good team autonomy. Tradeoffs: cloud lock-in risk and additional effort for heterogeneous source integration. Best fit: small-to-medium organizations needing fast platform enablement. BigQuery authorized views are an effective interface pattern for exposing a governed subset of data products to consumers. The producer keeps ownership of base tables while consumers query only approved views, which reduces accidental overexposure and simplifies access boundaries. Separation of concerns: producers manage source tables and interface contracts; consumers access stable SQL views.

Controlled access: authorized datasets and authorized views grant query access without direct table permissions on producer assets. Versioning strategy: interfaces can evolve through dataset versioning or view versioning, depending on compatibility and governance policy.

3.1 Consumer Producer Consumption

Operational impact: this model supports independent producer evolution while minimizing consumer disruption. The same interface approach can be extended to externalized data through BigLake tables, while maintaining SQL-based consumption contracts. Streaming data products can be exposed through Pub/Sub topic s, where each consumer creates independent subscriptions based on its own processing needs. This interface pattern reduces coupling between producer release cycles and consumer delivery pipelines, while preserving product ownership boundaries. Decoupled consumption: multiple consumers subscribe independently without requiring custom producer-side endpoints. Filtering and routing: subscription-level filtering supports consumer-specific streams from the same producer topic. Schema discipline: explicit schemas improve compatibility checks and reduce downstream parsing failures in high-volume flows.

Version evolution: breaking message changes are handled through new schemas and parallel subscriptions to limit disruption.
